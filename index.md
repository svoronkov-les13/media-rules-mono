# Правила сопоставления брифа и медиаплана

Документ описывает минимальную логику MVP для автоматизации digital-медиапланирования: как из клиентского брифа получить черновик медиаплана, который специалист может проверить, поправить и выгрузить в Excel.

MVP не заменяет медиапланера полностью. Он автоматизирует рутину: разбор вводных, проверку полноты, выбор сценария, первичный подбор площадок/форматов, расчёт сплита и генерацию Excel.

---

## 1. Вход и выход

### Вход: клиентский бриф

Система должна принимать брифы в разных формах:

- короткий текст из переписки;
- DOCX;
- PPTX;
- XLSX/табличные вводные;
- ручная web-форма.

Из брифа извлекаются поля:

| Поле | Пример | Назначение |
|---|---|---|
| Клиент / бренд | Client N, продукт, кампания | Заполнение шапки медиаплана |
| Период | месяц, даты старта/финиша | Расчёт длительности и флайта |
| Бюджет | с НДС / без НДС / с АК | Ограничение оптимизации |
| Гео | РФ, города, регионы, адреса | Фильтр площадок и таргетингов |
| ЦА | соцдем, интересы, сегменты | Выбор каналов и таргетингов |
| KPI | CPM, CPC, CPL, охват, лиды | Выбор формул и целевой функции |
| Каналы / форматы | VK, Яндекс, OLV, блогеры | Ограничения на подбор |
| Обязательства | цены из спецификации, лимит Smart TV | Валидация и hard constraints |
| Исторический контекст | по бенчам, по прошлым РК | Выбор бенчмарков и сценария |

### Выход: черновик медиаплана

Минимальный результат:

- Excel-файл в подходящем шаблоне;
- шапка кампании;
- строки размещений;
- бюджетный сплит;
- прогнозные показатели;
- расчёт НДС/АК/маржи;
- комментарии/предупреждения;
- список недостающих вводных.

## 2. Общий pipeline MVP

1. **Загрузка брифа** — пользователь загружает файл или вставляет текст.
2. **Извлечение вводных** — LLM/парсер приводит бриф к структурированному JSON.
3. **Валидация полноты** — система проверяет обязательные поля и формирует уточняющие вопросы.
4. **Классификация типа кампании** — определяется сценарий: performance, HR, OLV/banner, контекст, блогеры, retail media и т.д.
5. **Выбор шаблона медиаплана** — по типу кампании выбирается Excel-шаблон и набор вкладок.
6. **Подбор площадок и форматов** — на основе правил, справочников, бенчей и ограничений.
7. **Расчёт сплита и прогнозов** — распределение бюджета и расчёт показов/кликов/лидов/охвата/стоимости.
8. **Генерация Excel** — заполнение шаблона, формулы, вкладки, свод.
9. **Ручная проверка** — медиапланер меняет строки, ставки, бюджеты, KPI и пересчитывает итог.

## 3. Правила классификации брифа

| Условие в брифе | Тип кампании | Типовой выход |
|---|---|---|
| “вакансия”, “HR”, “кассир”, малый бюджет | HR / Lead Ads | VK LeadAds / соцсети, CPC/CPL |
| “заявка”, “CPL”, “форма”, “лендинг” | Performance / Lead generation | Яндекс Поиск, РСЯ, TG Ads, прогноз CPL |
| “OLV”, “баннеры”, “охват”, “видимость”, “фрод” | Медийная digital campaign | OLV/banner план, CPM, охват, viewability |
| “блогеры”, “Telegram”, “YouTube”, “подборка” | Influencer marketing | Вкладки блогеров, форматы пост/видео |
| “семантика”, “поиск”, “ключевые слова” | Контекст | Вкладки семантики, минус-фразы, CPC |
| Несколько объектов с отдельными бюджетами | Multi-object campaign | Отдельные строки/вкладки по объектам + итог |
| “маркетплейс”, “retail”, Ozon/WB | Retail media | Площадки retail media, CPM/CPC |

---

## 4. Правила выбора шаблона Excel

| Тип кампании | Минимальный шаблон |
|---|---|
| HR в соцсетях | VK/соцсети: одна вкладка с CPC/CPL и итогами |
| Контекст / объекты | Контекст + Итого: строки по объектам, бюджетам, CPC |
| Performance на заявки | МП + семантика + минус-фразы + источники данных |
| Медийная OLV/banner | Digital media plan: CPM, показы, охват, устройства, таргетинги |
| Блогеры | Digital + блогеры: реко, подборка, ссылки, форматы |
| Retail media | Digital основной + площадки + гео + аудитории |

Для MVP достаточно поддержать 3–5 шаблонов, покрывающих основные пары “бриф → медиаплан”. Остальные кейсы должны уходить в ручную доработку.

---

## 5. Правила выбора площадок и форматов

1. Если площадка явно указана в брифе — включить её первой, если бюджет и формат допустимы.
2. Если бюджет малый — не дробить его на много площадок.
3. Если KPI performance — приоритет каналам с измеримым действием: поиск, РСЯ, соцсети, LeadAds.
4. Если KPI охват/знание — приоритет OLV, programmatic, VK, video/display, Smart TV в рамках ограничений.
5. Если нужен B2B/финансовый продукт — добавить поиск, РСЯ, VK, тематические бизнес-площадки.
6. Если детская/родительская ЦА — добавить parenting/kids-сегменты, блогеров, VK, Яндекс, Avito при релевантности.
7. Если есть “цены из спецификации” — использовать только разрешённый прайс-лист.
8. Если площадка не поддерживает формат или гео — исключить её.

### Примеры

| Ситуация | Рекомендация |
|---|---|
| HR-вакансия, бюджет до 100k | VK LeadAds, простой план, прогноз лидов |
| Франшиза/заявки, РФ | Яндекс Поиск + РСЯ + TG Ads, CPL-прогноз |
| Имиджевая кампания, OLV/banner | Video/display, CPM, охват, ограничения по устройствам |
| Кампания с блогерами | Подборка Telegram/YouTube/VK по тематике и подписчикам |
| Несколько объектов с бюджетами | Каждому объекту отдельная строка и общий итог |

## 6. Правила расчётов

| Метрика | Формула |
|---|---|
| Стоимость без НДС | `budget_with_vat / 1.2` |
| Стоимость с НДС | `budget_without_vat * 1.2` |
| Показы при CPM | `budget / CPM * 1000` |
| Клики при CPC | `budget / CPC` |
| Клики через CTR | `impressions * CTR` |
| Лиды | `clicks * CR` |
| CPL | `budget / leads` |
| CPM факт/прогноз | `budget / impressions * 1000` |
| CPC факт/прогноз | `budget / clicks` |

### Бюджет и маржа

1. Все расчёты должны явно хранить режим бюджета: с НДС, без НДС, с АК/без АК.
2. Если задан минимальный порог маржинальности — строки ниже порога не добавляются или помечаются как риск.
3. Если прайс закупочный и клиентская цена различаются — медиаплан должен хранить обе цены и маржу.
4. Если бюджет задан “до X” — итог не должен превышать X.
5. Если задано несколько бюджетных опций — генерируются несколько версий плана.

---

## 7. Правила распределения бюджета

MVP использует не “магический ML”, а прозрачный rule-based сплит.

| Тип задачи | Базовая логика сплита |
|---|---|
| Performance / заявки | 50–70% в наиболее конверсионный канал, остаток в расширение охвата/ретаргетинг |
| HR | 80–100% в канал с LeadAds/заявками, если бюджет малый |
| Контекст с объектами | Бюджет распределяется по объектам из брифа без изменения, если не сказано иначе |
| Медийная кампания | Бюджет распределяется по форматам с контролем охвата, частоты, устройств |
| Блогеры | Бюджет распределяется по списку обязательных и рекомендованных блогеров |
| Retail media | Распределение по селлерам/форматам с учётом гео и доступного прайса |

Отклонения от правил должны быть видны пользователю как комментарии: “выбрано из-за KPI CPL”, “исключено из-за бюджета”, “требуется ручная проверка цены”.

## 8. Правила валидации брифа

### Hard-stop ошибки

Без этих данных нельзя корректно посчитать план:

- нет бюджета;
- нет периода;
- нет типа продукта/кампании;
- неизвестно, с НДС бюджет или без;
- нет целевого действия для performance-кампании;
- нет гео, если площадки требуют географического таргетинга.

### Warning-ошибки

Можно посчитать черновик, но нужна проверка:

- нет KPI;
- нет посадочной страницы;
- нет исторических бенчей;
- нет фактических данных прошлых РК;
- неясна ЦА;
- задана площадка, но нет формата;
- указан “по бенчам”, но источник бенчей не приложен.

## 9. Минимальная структура данных

### Brief JSON

```json
{
  "client": "Client N",
  "campaign": "Product / campaign name",
  "period": { "from": "YYYY-MM-DD", "to": "YYYY-MM-DD" },
  "budget": { "amount": 1000000, "vat": "included", "agency_commission": "unknown" },
  "geo": ["RF"],
  "audience": ["description / segment"],
  "kpi": ["CPL", "CPM", "reach"],
  "mandatory_channels": ["VK", "Yandex"],
  "constraints": ["use contract specification prices"],
  "source_notes": ["according to benchmarks and past campaigns"]
}
```

### MediaPlan Row

```json
{
  "platform": "Yandex Direct",
  "placement": "Search / RSYA",
  "format": "text / banner / video",
  "buying_unit": "CPC",
  "period": "01.09-30.09",
  "budget_without_vat": 100000,
  "budget_with_vat": 120000,
  "price": 50,
  "impressions": 100000,
  "clicks": 2000,
  "ctr": 0.02,
  "cr": 0.03,
  "leads": 60,
  "comment": "selected by performance rule"
}
```

## 10. Что не входит в минимальный MVP

Минимальное исполнение не должно обещать:

- полноценный Data Lake;
- real-time API всех рекламных систем;
- автоматическую проверку свободного инвентаря;
- автобронирование;
- самостоятельное ML-прогнозирование без исторических fact-data;
- универсальное покрытие всех Excel-шаблонов;
- принятие финального решения без медиапланера.

Это следующий этап после MVP.

---

## 11. Критерий готовности MVP

MVP можно считать успешным, если на исторических парах “бриф → медиаплан” система:

1. корректно извлекает основные вводные;
2. выбирает правильный тип кампании;
3. выбирает подходящий шаблон;
4. генерирует черновик Excel;
5. считает бюджет и прогнозные метрики без арифметических ошибок;
6. явно показывает, где требуются ручные решения;
7. сокращает время подготовки первичного медиаплана.

---

## 12. Роль человека

Медиапланер остаётся в контуре:

- проверяет корректность интерпретации брифа;
- подтверждает площадки и цены;
- правит сплит;
- утверждает финальный Excel;
- добавляет экспертные комментарии.

Это важно для пресейла: MVP — это ассистент, а не автономная система закупки рекламы.
