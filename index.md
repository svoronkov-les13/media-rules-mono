# БРИФ → МЕДИАПЛАН

Разбор исторических пар `Бриф_clientN → МП_clientN` и правила для MVP автоматизации digital-медиапланирования.

Минимальный продукт: ассистент медиапланера. Он разбирает бриф, проверяет вводные, выбирает сценарий/шаблон, предлагает площадки и сплит, считает прогнозы и формирует Excel-черновик. Финальное решение остаётся за специалистом.

[Скачать редактируемый DOCX-файл](exports/media-rules-export.docx)

## Сквозные правила

1. **Бриф сначала нормализуется в JSON**, а не сразу в Excel.
2. **Тип кампании определяет шаблон**: HR, контекст, performance/CPL, OLV/banner, блогеры, retail media, multi-object.
3. **Excel-шаблон выбирается до расчётов**, потому что структура вкладок зависит от типа задачи.
4. **Справочники обязательны**: площадки, форматы, ставки, НДС/АК, гео, ЦА, техтребования, договорные цены.
5. **Исторические пары — не ML-датасет**, а база правил, few-shot примеров и regression-тестов.
6. **Неполные вводные нельзя молча додумывать**: система должна показывать допущения и warning/hard-stop.
7. **Медиапланер остаётся в контуре**: правит сплит, ставки, площадки и утверждает финальный Excel.

## Что автоматизируется в MVP

- извлечение параметров из TXT/DOCX/PPTX;
- проверка полноты брифа;
- выбор типа кампании;
- выбор Excel-шаблона;
- первичный подбор площадок/форматов;
- расчёт бюджета, НДС/АК, CPM/CPC/CPL, показов/кликов/лидов/охвата;
- генерация Excel-черновика;
- список допущений и ошибок.

## Что не обещать в MVP

- полностью автономный “идеальный” медиаплан;
- real-time API всех площадок;
- ML-прогноз без фактических данных;
- автоматическую проверку инвентаря;
- универсальную поддержку всех Excel-вариантов без шаблонизации.

## Реалистичный прототип

Рабочий прототип можно сделать вполне быстро. Минимальная цель: **бриф → нормализованный JSON**, пригодный для следующего шага: подбор шаблона, площадок и бюджетного сплита.

### MVP-прототип

**Вход:**

- `.txt`;
- `.docx`;
- `.pptx`;
- позже `.pdf`;
- текст из формы или Telegram.

**Выход:** структурированный JSON с извлечёнными вводными, классификацией кампании, missing fields, warnings и рекомендуемым следующим шагом.

Пример:

```json
{
  "client": "Client4",
  "campaign": "Франшиза пекарни с кредитной поддержкой",
  "product_category": "финансовый продукт / франшиза",
  "campaign_type": "performance_leadgen",
  "period": {
    "raw": "октябрь",
    "start": null,
    "end": null
  },
  "budget": {
    "amount": 1000000,
    "currency": "RUB",
    "vat_mode": "included",
    "agency_commission": "unknown"
  },
  "geo": ["Россия"],
  "audience": [
    "интересующиеся открытием бизнеса",
    "потенциальные франчайзи"
  ],
  "kpi": ["CPL", "leads"],
  "target_action": "заявка в форме",
  "channels_requested": [],
  "channels_suggested": ["Яндекс Директ", "РСЯ", "Telegram Ads"],
  "formats": ["search ads", "TGO", "banner", "text+link"],
  "constraints": [
    "нужен расчет CPL",
    "предмет рекламы — франшиза с поддержкой банка"
  ],
  "missing_fields": [
    "посадочная страница",
    "целевой CPL",
    "исторический CR"
  ],
  "warnings": [
    "CR и CPL можно оценить только по бенчам"
  ],
  "next_step": "generate_media_plan_draft"
}
```

### Что будет внутри

1. **Парсер файлов** — `txt/docx/pptx → text`.
2. **LLM extractor** — извлекает поля по JSON Schema.
3. **Rule classifier** — определяет тип кампании: HR, performance/CPL, context, OLV/banner, influencer, retail media, multi-object, historical planning.
4. **Validator** — разделяет hard-stop и warning: нет бюджета, периода, гео, KPI, неясен НДС, указано “по бенчам”, но бенчи не приложены.
5. **Normalizer** — приводит бюджет, даты, гео, каналы, KPI и форматы к единому виду.
6. **Output** — JSON, человекочитаемый summary, позже Excel draft.

### Оценка сроков

**Очень быстрый прототип: 2–4 дня**

- CLI или простой web endpoint;
- загрузка `txt/docx/pptx`;
- JSON extraction;
- классификация;
- warnings/missing fields;
- прогон на 10 исторических брифах.

**Нормальный demo-прототип: 1–2 недели**

- небольшой web-интерфейс;
- upload файла;
- просмотр extracted JSON;
- сравнение с эталонными парами;
- экспорт JSON;
- начальный выбор шаблона медиаплана;
- сохранение результатов.

**До черновика медиаплана: 3–5 недель**

- справочник площадок;
- правила подбора;
- расчёт базовых KPI;
- генерация Excel по 2–3 шаблонам.

### Presale-формулировка

> За 1–2 недели можно сделать demo-прототип, который принимает реальный бриф и возвращает структурированный JSON с классификацией кампании, вводными, недостающими полями и рекомендацией следующего шага.

Это реалистичный первый артефакт перед генерацией медиаплана.

## Оценка этапов проекта

Оценка разделена на три самостоятельных этапа. Каждый этап можно запускать отдельно, с отдельной приёмкой результата.

Базовая расчётная ставка: **4 000 ₽/час**. В ставку заложены разработка, аналитика, управление, тестирование, DevOps, документация и проектные риски.

| Этап | Срок | Часы | Стоимость | Результат |
|---|---:|---:|---:|---|
| 1. Аналитика | 3–4 недели | 600 ч | 2,4 млн ₽ | требования, схема данных, правила классификации, backlog, JSON Schema |
| 2. Разработка MVP | 8–10 недель | 1 725 ч | 6,9 млн ₽ | загрузка брифов, extraction в JSON, классификация, validation, warnings, базовый UI/API |
| 3. Полное решение | 14–18 недель | 2 550 ч | 10,2 млн ₽ | справочники, правила подбора, расчёты KPI, Excel-генерация, роли, хранение, журналирование |
| **Итого** | **25–32 недели** | **4 875 ч** | **19,5 млн ₽** | полный цикл от аналитики до рабочего решения |

### Команда и роли

| Роль | Участие | Зона ответственности |
|---|---:|---|
| Product / бизнес-аналитик | 0,5–1 FTE | требования, сценарии, приёмка, backlog, коммуникация с медиапланерами |
| Системный аналитик | 0,5–1 FTE | JSON Schema, правила, валидации, модели данных, интеграционные контракты |
| Tech Lead / архитектор | 0,3–0,7 FTE | архитектура, технические решения, ревью, качество реализации |
| Backend-разработчик | 1–2 FTE | API, хранение, парсинг файлов, бизнес-логика, расчёты |
| Frontend-разработчик | 0,5–1 FTE | web-интерфейс, upload, просмотр JSON, редактирование, экспорт |
| ML/LLM engineer | 0,5–1 FTE | extraction, prompts, JSON Schema, качество распознавания, regression-тесты |
| QA engineer | 0,3–1 FTE | тест-кейсы, проверка исторических брифов, регрессия, приёмочные проверки |
| DevOps engineer | 0,2–0,5 FTE | окружения, деплой, мониторинг, секреты, CI/CD |
| Project manager | 0,3–0,5 FTE | план, риски, сроки, статус, координация команды |

FTE указаны ориентировочно: на разных этапах состав команды меняется. Например, на аналитике больше аналитиков, на MVP больше backend/LLM, на полном решении добавляются QA, DevOps и frontend.

### 1. Аналитика

**Цель:** зафиксировать, что именно автоматизируем, какие типы брифов поддерживаем, какие шаблоны медиапланов нужны в первую очередь и какие ограничения нельзя нарушать.

**Срок:** 3–4 недели.  
**Оценка:** 600 ч × 4 000 ₽ = **2,4 млн ₽**.

| Блок работ | Длительность | Роли | Часы | Стоимость |
|---|---:|---|---:|---:|
| Разбор исторических пар “бриф → медиаплан” | 1–1,5 недели | BA, системный аналитик, медиапланинг SME | 120 ч | 480 тыс. ₽ |
| Описание типов кампаний и правил классификации | 1 неделя | BA, системный аналитик, LLM engineer | 100 ч | 400 тыс. ₽ |

| Описание hard-stop/warning validation | 0,5–1 неделя | BA, системный аналитик, QA | 80 ч | 320 тыс. ₽ |
| Backlog MVP и полного решения | 0,5 недели | Product, Tech Lead, PM | 80 ч | 320 тыс. ₽ |
| Критерии приёмки, оценка рисков, документация | 0,5 недели | Product, PM, Tech Lead | 100 ч | 400 тыс. ₽ |
| **Итого** | **3–4 недели** |  | **600 ч** | **2,4 млн ₽** |

**Результат:** аналитический пакет, достаточный для разработки без постоянного пересогласования логики.

### Коммерческая логика

Бюджет полного цикла — **19,5 млн ₽**. Он считается по единой ставке **4 000 ₽/час** и суммарной трудоёмкости **4 875 часов**. Это оставляет запас до 20 млн ₽ и снижает риск недооценки: аналитика закрывает неопределённость, MVP быстро даёт проверяемый результат, полный этап добавляет расчёты, справочники и production-качество.
